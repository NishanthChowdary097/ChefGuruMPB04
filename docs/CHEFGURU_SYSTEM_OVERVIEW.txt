ChefGuru System Architecture and Services Overview

This document provides a comprehensive overview of the ChefGuru application, which is a recipe generation and management platform that leverages artificial intelligence to help users create culinary recipes based on available ingredients. The system is built using modern web development practices with a microservices architecture, allowing for independent scaling and deployment of different components.

At its core, ChefGuru is designed to solve a real problem: users often wonder what they can cook with the ingredients they have on hand. By breaking down the system into separate services that handle authentication, business logic, and user interface independently, we've created a scalable foundation that can grow with the platform's needs. The entire system runs in Docker containers, which means the same application can run on a developer's laptop or in a production data center without any changes to the code.

The system is composed of five main components working together. First, there's the Authentication Service running on port 5000, which handles all user-related operations like registration, login, and password reset. Then there's the Application Service on port 6000, which contains the actual recipe generation logic and integrates with artificial intelligence services. The frontend is a React-based web application served on port 4000 that provides the user interface. Behind the scenes, MongoDB stores all persistent data including user accounts, recipes, and ingredients, while Redis handles caching and rate limiting to keep the system performant and protected from abuse.

The authentication flow is fairly straightforward but robust. When someone first visits ChefGuru, they need to create an account. They provide their username, email address, and password. The Authentication Service receives this information, checks that the email isn't already in use, creates a new user record in MongoDB, and sends a verification email. This email contains a special link that the user must click to verify they actually own that email address. Once verified, they can log in. When logging in, the service checks their credentials, and if everything is correct, it generates two tokens: an access token that expires relatively quickly and a refresh token that lives longer. These tokens are JWT tokens, which is a standard way to handle authentication in web applications. The beauty of JWT is that the server doesn't need to remember anything about the user; it just needs to know the secret key used to create the token, and it can verify that the token is legitimate.

Password reset works in a similar but slightly different way. If a user forgets their password, they request a reset, and the service sends them an email with a unique token. This token is only valid once and only for a limited time. When they click the link in the email, they can set a new password. This approach ensures that even if someone sees the reset email, they can't misuse it after the password has already been changed.

The User model in the database is quite simple. It stores the username, email address, and hashed password. It also keeps track of whether the email has been verified and maintains a reference to any saved recipes the user has created. The timestamp of when the account was created is also recorded, which is useful for analytics and understanding user growth. The Mail model is separate and specifically handles the tokens and codes used in the verification and password reset workflows. This separation makes sense because mail-related data has different lifecycle requirements than user data.

Once a user is logged in, they interact with the Application Service, which is where the real magic happens. This service has two main endpoints: one for generating recipes and another for getting detailed explanations of cooking steps. To generate a recipe, a user provides a list of ingredients they have available, up to a maximum of fifteen ingredients. The service takes this list, sends it to an external AI service hosted on HuggingFace spaces, and gets back a complete recipe with a title, description, ingredients list, cooking instructions with timing information, and estimated prep and cooking times.

The recipe generation process has some clever optimizations built in. First, the service integrates with the HuggingFace API, which is where the actual AI model lives. If that service is down or slow, the code will retry up to three times before giving up, with a one-second delay between attempts. This resilience is important because external services aren't always perfectly reliable. The entire request times out after forty-five seconds, which prevents the system from hanging indefinitely if something goes wrong.

When a recipe is generated, something interesting happens. Instead of immediately saving it to the database, the system stores it temporarily in Redis for fifteen days. This gives users time to review the generated recipe, try it out if they want, and then decide whether to save it permanently or discard it. Meanwhile, all the ingredients mentioned in the recipe are automatically catalogued in the ingredients collection, with duplicates eliminated. This means over time, the system builds up a comprehensive list of ingredients that have been used in recipes, which can be used for filtering and suggestions.

The Recipe model is quite comprehensive and reflects the complexity of cooking. It stores the title and description, the user who created it, whether it's marked as public, a list of ingredients with quantities, step-by-step instructions with timing information, the prep time, cook time, number of servings, and the original prompt that was used to generate it if it came from the AI. The instructions are structured as a series of steps, each with a step number, the actual instruction text, timing, description, and potentially an explanation dictionary. This structure allows for future enhancements where users could ask for explanations of specific steps, and those explanations would be stored right alongside the step.

The frontend is a single-page application built with React, which means it loads once and then dynamically updates as the user interacts with it. The routing is handled by React Router, so when users go to different URLs, they stay on the same application but see different content. The application has a landing page for first-time visitors, login and signup pages for authentication, and a main application page where authenticated users can generate and manage recipes. The authentication state is managed through React Context, which is a way to share data across components without having to pass it through every intermediate component.

The frontend communicates with the backend services exclusively through HTTP requests. It sends JSON data and expects JSON responses back. When the backend returns an error, especially a 401 unauthorized error, the frontend knows the session has expired and redirects the user back to the login page. This seamless handling of session expiration provides a good user experience because users don't get confused about why they suddenly can't access their recipes.

Rate limiting is an important but often overlooked aspect of web services. ChefGuru implements rate limiting to protect against abuse and to ensure fair usage. The signup endpoint limits users to three attempts per minute from the same IP address, which prevents someone from trying to create hundreds of accounts automatically. Login is limited to five attempts per minute, which is enough for someone to enter their credentials a few times if they mistype but prevents brute force attacks. The health check endpoints are limited to ten requests per minute. These limits are enforced using Redis, which keeps track of how many requests have been made from each IP address in each time window. If someone exceeds the limit, they get a 429 Too Many Requests response telling them to try again later.

The error handling throughout the system is consistent and helpful. Whether a user gets a 400 bad request error, a 401 unauthorized error, a 403 forbidden error, a 404 not found error, a 429 rate limit error, or a 500 internal server error, they get a JSON response with an error type and a human-readable message. This consistency makes it easier to handle errors in the frontend and makes debugging easier for developers. All errors are also logged using structlog, a structured logging library that captures not just the error message but also context like which user triggered the error, which service the error occurred in, and what other data might be relevant to understanding what went wrong.

The configuration system allows different settings for development and production environments. During development, the API documentation is available at the /docs endpoint, which is useful for testing and understanding the API. In production, this documentation is disabled for security reasons. Database connections are configured through environment variables, which makes it easy to point to different databases in different environments without changing code. The same goes for email configuration, Redis configuration, and all the other settings that might differ between environments.

Docker and Docker Compose orchestrate all the services. The docker-compose.yaml file defines all the containers that need to run and how they connect to each other. This means developers can run the entire application stack with a single command and don't need to worry about installing MongoDB, Redis, or any other dependency on their machine. Each service has its own Dockerfile that specifies how to build its container image. The images are built once and then run identically on any machine that has Docker installed, which largely eliminates the "it works on my machine" problem that has plagued software development forever.

Behind the scenes, logging is incredibly detailed. Every request that comes in is logged, every error is captured, every database query is recorded. This wealth of information is invaluable when trying to debug issues in production. The logs are structured, meaning they're stored as JSON objects with consistent fields like timestamp, log level, service name, and the actual message. This structure makes it possible to search and filter logs programmatically, not just by reading through them manually.

The system is designed to be easily extended and scaled. As the number of users grows, additional instances of the Authentication Service and Application Service can be started behind a load balancer. The database can be scaled with replica sets and sharding. The Redis instance can be scaled into a cluster. The frontend can be served from a content delivery network. All of these scaling options become viable because the services are independent and stateless, meaning they don't rely on being the only instance of their service running.

Looking toward the future, there are several enhancements that would improve the system. An API Gateway could sit in front of the services and handle routing, authentication, and rate limiting in a centralized place. A message queue like RabbitMQ could handle asynchronous tasks like sending emails, so the user doesn't have to wait for the email to actually be sent to get a response from the API. Kubernetes could orchestrate the containers instead of Docker Compose, providing better scheduling, self-healing, and resource management. A distributed tracing system like Jaeger could show the path that a request takes through all the services, making it easier to find performance bottlenecks. A service mesh like Istio could handle retries, circuit breaking, and load balancing automatically without code changes.

The project is organized so that developers can work independently on different parts. The auth folder contains all authentication-related code. The main folder contains the application service code. The frontend folder contains the React application. The ODM folder contains the data models that are used by the services. Each developer can work on their assigned area without stepping on other developers' toes, and changes can be merged in regularly.

When someone new joins the project, they need to understand the overall architecture but not necessarily the details of every service. The microservices approach makes this easier because each service is self-contained. A frontend developer might never need to look at the authentication service code. A backend developer might never need to look at the React components. But everyone benefits from understanding how the pieces fit together and what the responsibilities of each service are.

Security is baked in throughout the system. Passwords are stored in the database, though in a real production system they would be hashed. Email addresses must be unique to prevent multiple accounts with the same email. Email verification ensures that accounts are not created with fake email addresses. Password reset tokens are single-use and time-limited. JWT tokens expire and require refresh. CORS is configured so that the API only accepts requests from allowed origins. Rate limiting prevents brute force attacks and protects the system from being overwhelmed by requests.

The caching strategy strikes a balance between performance and consistency. Temporary recipes are cached in Redis for fifteen days, which is long enough that a user would expect their recipe to still be there if they come back to it, but not so long that the cache wastes memory or becomes inconsistent with the database if the recipe is deleted. The ingredients list is cached as a set in Redis for quick lookup, which is important because the system checks the ingredients list every time a recipe is generated. These caching decisions have been made with thought to the tradeoffs between memory usage, CPU usage, and consistency.

The codebase follows well-established design patterns. The Factory Pattern is used to create Flask applications with injected configuration, which makes testing easier and allows different configurations to be used in different environments. Blueprints are used to organize related routes, which keeps the code organized and makes it easier to find related functionality. The service locator pattern is used for some global services like the database and cache connections, which is a pragmatic choice that simplifies the code without creating excessive coupling.

Looking at the API endpoints, they're designed with REST principles in mind. Signing up is a POST request because it creates something new. Logging in is a POST request because it initiates an action. Resetting a password is a PUT request on the reset-pwd endpoint, which is appropriate for modifying something. Getting a recipe explanation is a POST request because it's a computational action that requires input data. The endpoints use appropriate HTTP status codes: 200 for success, 201 for creation, 400 for bad requests, 401 for authentication failures, 404 for not found, 429 for rate limiting, and 500 for server errors.

The system achieves a good balance between flexibility and constraints. Ingredients lists can be up to fifteen items, which is reasonable but prevents someone from making absurdly large requests. Email verification is required before login, which ensures that accounts are real. Passwords must be at least six characters, which is a minimum security requirement. Recipes can be marked as public or private, which gives users control over their data. These constraints are not arbitrary; they serve purposes ranging from security to preventing abuse to improving the user experience.

One of the clever aspects of the system is how it handles temporary recipes. Most applications would require a user to immediately save a recipe to the database after generation. But ChefGuru allows users to keep a recipe in their browser or in the Redis cache while they decide whether they actually want to cook with it. This is a user experience improvement that makes the system feel more responsive and less pushy. Users can generate recipes quickly without committing them to their personal recipe collection.

The system's logging infrastructure is set up for observability in production. By looking at logs, developers and operations teams can see what's happening in the system. They can identify trends, spot errors, and understand user behavior. The structured logging approach means that logs can be indexed and searched programmatically, not just read by humans. There's even infrastructure in place for Prometheus to scrape metrics, Loki to aggregate logs, and Grafana to visualize everything, though these might not be fully configured in the current deployment.

The way the services communicate is important to understand. The frontend talks to the backend services via HTTP. The backend services don't talk to each other directly; they only communicate through the database and cache. The Authentication Service might create a user in MongoDB, and the Application Service reads that user to validate that a request is from an authenticated user, but they never make HTTP calls to each other. This loose coupling is by design and makes the system more resilient. If the Authentication Service goes down, the Application Service can still serve recipes to users who are already logged in because the JWT token is self-contained and doesn't require contacting the Authentication Service to validate.

The choice of technologies throughout the system reflects a pragmatic approach to building web applications. Flask is lightweight and popular for Python web services. React is the most widely used JavaScript frontend framework. MongoDB is a popular choice for applications that need flexible schemas and quick iteration. Redis is the de facto standard for caching and rate limiting. Docker is the industry standard for containerization. These choices mean that developers who have worked on other projects will be familiar with the technologies, making it easier to hire and onboard new team members.

When thinking about the future of ChefGuru, it's worth considering how the system will evolve. As the user base grows, the current single-instance setup will eventually hit bottlenecks. The database will need to handle more concurrent queries. The cache will need to be bigger. The services will need to be replicated. All of these scaling challenges are well-understood problems with known solutions, and the microservices architecture makes these solutions straightforward to implement. The hard work of building a system that works at all is complete; scaling it to handle more users is mostly an engineering and operational challenge at that point.

In conclusion, ChefGuru demonstrates how a thoughtful architecture can enable a small team to build a system that is not only functional but also scalable, maintainable, and pleasant to work on. The separation of concerns into different services, the careful attention to security, the use of industry-standard technologies, the containerization approach, and the attention to details like error handling and logging all contribute to a system that can grow as the product grows. While there are always ways to improve any system, ChefGuru's foundation is solid and positions the project well for future success.
