EXECUTIVE SUMMARY
================================================================================

ChefGuru is a modern, distributed microservices-based application designed to
generate culinary recipes using artificial intelligence and manage user-
generated content. The system employs a modular architecture with independent
authentication and application services, coupled with a React-based frontend,
MongoDB for persistent storage, and Redis for caching and rate limiting.

This documentation provides an in-depth analysis of all major services,
their responsibilities, data models, API endpoints, and architectural patterns
employed throughout the system. The architecture follows industry-standard
practices including Factory Pattern, WSGI compliance, and distributed
caching strategies.


================================================================================
SECTION 1: SYSTEM ARCHITECTURE OVERVIEW
================================================================================

1.1 ARCHITECTURAL PARADIGM

The ChefGuru system implements a **Microservices Architecture** with the
following characteristics:

* **Service Independence**: Each microservice operates independently with
  separate ports and deployment containers
* **Separation of Concerns**: Authentication logic is isolated from business
  logic
* **Scalability**: Individual services can be scaled without affecting others
* **Technology Agnostic**: Different services can use different technology
  stacks if required
* **Containerization**: All services are containerized using Docker for
  consistent deployment

1.2 CORE COMPONENTS

The system comprises five primary components:

1. **Authentication Service** (Port 5000)
   - Handles user registration, login, and credential management
   - Issues JWT tokens for secure API access
   - Manages email verification and password reset workflows

2. **Application Service** (Port 6000)
   - Core business logic layer
   - Recipe generation and management
   - Integration with AI/ML backend
   - User data persistence and retrieval

3. **Frontend Application** (Port 4000)
   - React-based single-page application
   - Built with Vite for optimal development experience
   - Client-side routing and state management
   - User interface for recipe discovery and management

4. **MongoDB Database**
   - Persistent data storage
   - Document-oriented NoSQL database
   - Authentication via username and password
   - Default port: 27017

5. **Redis Cache**
   - In-memory data cache for performance optimization
   - Rate limiting storage
   - Temporary recipe cache for unconfirmed generations

1.3 COMMUNICATION FLOW

The typical user interaction flow follows this pattern:

User Interface (Frontend) 
    ↓
HTTP/REST Requests
    ↓
Authentication Service (validates credentials, issues tokens)
    ↓ (with JWT in headers)
Application Service (processes business logic)
    ↓
Database/Cache Layer (MongoDB & Redis)
    ↓
Response back through Application Service
    ↓
Return to Frontend

2.1 SERVICE PURPOSE

2.2 KEY RESPONSIBILITIES

* JWT token generation (access and refresh tokens)
* Password reset request processing
* Token refresh and validation
* Rate limiting for security

2.4 API ENDPOINTS

**Authentication Routes Base (/api/auth/):**

1. **Health Check - GET /api/auth/**
   - Purpose: Verify service availability
   - Rate Limit: 10 requests per minute
   - Response: {"status": "alive", "server": "auth.routes.base"}
   - Authentication: Not required

2. **User Registration - POST /api/auth/user/signup**
   - Purpose: Create new user account
   - Rate Limit: 3 requests per minute
   - Required Fields: username, email, password
   - Validation: Email uniqueness, password strength
   - Response: User object with ID, username, email, created_at
   - Post-Action: Sends verification email via Mail service

3. **User Login - POST /api/auth/user/login**
   - Purpose: Authenticate user and issue tokens
   - Rate Limit: 5 requests per minute
   - Required Fields: email, password
   - Validation: User existence, password match, email verification
   - Response: Access token, refresh token
   - On Failure: Returns "Invalid credentials" or "Please verify your email"

4. **Password Reset Request - POST /api/auth/user/reset-pwd**
   - Purpose: Initiate password reset workflow
   - Rate Limit: 3 requests per minute
   - Required Fields: email
   - Response: Always returns success message (for security)
   - Post-Action: If user exists, sends reset email with token and code

5. **Password Reset Confirmation - PUT /api/auth/user/reset-pwd**
   - Purpose: Complete password change after verification
   - Required Fields: user_id, token, password
   - Validation: Token validity, token not previously used
   - Response: Success message
   - State Change: Updates user password, marks token as used

6. **Email Verification - GET /api/auth/mail/verify/:userId/:token**
   - Purpose: Verify user email address
   - Validation: Token existence and validity
   - Response: Renders verification template or error page
   - State Change: Sets user.is_verified to True

2.5 DATA MODELS - USER

The User model in MongoDB contains:

```
User {
    username: String (required)
    email: EmailField (required, unique)
    password: String (required, min 6 characters)
    is_verified: Boolean (default: False)
    saved_recipes: [References to SavedRecipe]
    created_at: DateTime (default: current UTC time)
}
```

2.6 DATA MODELS - MAIL

The Mail model manages email communication tokens:

```
Mail {
    user_id: Reference to User (required)
    type: String in ["verification", "reset"] (default: "verification")
    code: String (6-digit random code)
    token: UUID (unique identifier for link)
    is_used: Boolean (default: False)
    created_at: DateTime (default: current UTC time)
}
```

2.7 SECURITY FEATURES

**JWT Configuration:**
* Algorithm: HS256 (HMAC with SHA-256)
* Token Location: Headers and Cookies
* Leeway: 15 seconds for clock skew tolerance
* CORS Handling: Configured with allowed origins
* CSRF Protection: Enabled for cookies

**Rate Limiting:**
* Signup: 3 attempts per minute
* Login: 5 attempts per minute
* Health Check: 10 attempts per minute
* Default Rate: 200 requests per minute per IP

**Error Handling:**
* Centralized error handler registration
* HTTP status codes: 400, 401, 403, 404, 429, 500
* JSON response format standardization
* Detailed logging of errors

SECTION 3: APPLICATION SERVICE DETAILED ANALYSIS
================================================================================

3.1 SERVICE PURPOSE

The Application Service contains core business logic, specifically recipe
generation, management, and retrieval. It serves as the primary API layer
for authenticated users and integrates with external AI services.

3.2 KEY RESPONSIBILITIES

* Recipe generation from ingredient lists
* Step-by-step recipe explanation
* Ingredient management and tracking
* Recipe storage and retrieval
* User-specific recipe history
* Integration with HuggingFace AI API
* Caching of temporary recipes

3.3 APPLICATION INITIALIZATION

Initialization mirrors the Authentication Service with identical patterns
for configuration, logging, database, and cache initialization.

3.4 API ENDPOINTS

**Recipe Routes (/api/app/recipe/):**

1. **Generate Recipe - POST /api/app/recipe/generate**
   - Purpose: Create new recipe from ingredient list
   - Authentication: Required (JWT)
   - Rate Limit: 5 requests per minute
   - Required Fields: ingredients (array of strings)
   - Validation: Max 15 ingredients per request
   - Response: Recipe object with generated content
   - Post-Action: Stores temporarily in Redis (15-day expiration)

2. **Explain Recipe Step - POST /api/app/recipe/explain_step**
   - Purpose: Get detailed explanation for specific recipe step
   - Authentication: Required (JWT)
   - Required Fields: step_number, context, recipe_id
   - Validation: Step number within recipe bounds
   - Response: Explanation dictionary with AI-generated content
   - Data Source: Either from saved recipe or temporary cache

3.5 AI INTEGRATION

The service integrates with an external HuggingFace AI endpoint for recipe
generation:

**Endpoint:** Configured via Config.HF_API (default: https://mosako-chef-guru.hf.space)

**Retry Logic:**
* Attempts: 3 retries with 1-second delay between attempts
* Timeout: 45 seconds per request
* Fallback: Raises exception if all attempts fail

**Recipe Generation Payload:**
```
{
    "ingredients": ["ingredient1", "ingredient2", ...]
}
```

**Response Structure:**
```
{
    "title": String,
    "description": String,
    "ingredients": [String],
    "instructions": [
        {
            "step": String,
            "timing": String,
            "description": String
        }
    ],
    "prep_time": Integer,
    "cook_time": Integer,
    "servings": Integer
}
```

3.6 DATA MODELS - INGREDIENT

```
Ingredient {
    name: String (required, unique, indexed)
    unit: String (optional, e.g., "grams", "cups", "tsp")
    calories_per_unit: Float (optional)
}
```

Methods:
* to_dict(): Converts document to dictionary with id, name, unit, calories

3.7 DATA MODELS - RECIPE INGREDIENT

Embedded document within Recipe:

```
RecipeIngredient {
    ingredient: Reference to Ingredient (required)
    quantity: String (required, e.g., "2 cups")
}
```

3.8 DATA MODELS - RECIPE STEP

Embedded document within Recipe:

```
RecipeStep {
    step_number: Integer
    step: String (cooking instruction)
    timing: String (time required)
    description: String (detailed description)
    explanation: Dictionary (user-requested explanations)
}
```

3.9 DATA MODELS - RECIPE

```
Recipe {
    user: Reference to User (required)
    title: String (required)
    description: String
    public: Boolean (default: False)
    ingredients: [RecipeIngredient] (embedded documents)
    instructions: [RecipeStep] (embedded documents)
    prep_time: Integer (minutes)
    cook_time: Integer (minutes)
    servings: Integer
    generation_prompt: String (original prompt used)
    created_at: DateTime (indexed, default: current UTC time)
}
```

Database Indexes:
* user: For quick retrieval by user
* -created_at: Reverse chronological ordering

3.10 CACHING STRATEGY

Temporary recipes utilize Redis with the following pattern:

**Key Format:** `recipe_temp:{user_id}:{recipe_id}`
**Storage:** JSON serialization of complete recipe object
**TTL:** 15 days (1,296,000 seconds)
**Purpose:** Allow users to generate recipes without immediate persistence

This permits users to generate recipes, review them, and decide whether to
save them permanently to the database.

3.11 INGREDIENT MANAGEMENT

The system automatically manages ingredient cataloging:

* When a recipe is generated, ingredients are extracted
* Each ingredient is checked for existence in the database
* New ingredients are created automatically if not found
* Ingredient names are normalized (lowercase, trimmed)
* Ingredients are indexed in Redis set for quick lookup

4.5 AUTHENTICATION CONTEXT

The AuthContext provides global authentication state:

```
Context Properties:

- login(): Authenticate user
- logout(): Terminate user session
- signup(): Register new user
- updateAuth(): Refresh auth state
```

The context uses JWT tokens stored in localStorage for session persistence
across page refreshes.

SECTION 5: DATA PERSISTENCE AND CACHING
================================================================================

5.1 MONGODB CONFIGURATION

**Connection Details:**
* Default Host: localhost
* Default Port: 27017
* Authentication: Username/Password
* Format: mongodb://{username}:{password}@{host}:{port}/{database}?authSource=admin

**Collections:**
* users: User account data
* recipes: User-generated recipes
* ingredients: Ingredient catalog
* mails: Email verification and reset tokens

**Indexing Strategy:**
* Single field indexes on frequently queried fields
* Compound indexes for multi-field queries
* TTL indexes not explicitly used (handled by application logic)

5.2 REDIS CONFIGURATION

**Connection Details:**
* Default Host: localhost
* Default Port: 6379
* Optional Authentication: password
* Database Separation:
  * DB 0: Application data cache (recipe_temp)
  - DB 1: Rate limiting storage

**Data Structures Used:**
* Strings: Recipe JSON serialization
* Sets: Ingredient lookup cache
* Sorted Sets: Potential future use for rate limiting
* Hashes: Session storage

5.3 CACHE INVALIDATION

**Recipe Cache:**
* TTL-based expiration (15 days)
* Manual deletion on recipe save to database
* Manual deletion on corrupted data detection

**Rate Limiting:**
* Redis-based sliding window
* Automatic cleanup by Redis
* Per-IP address tracking

**Ingredient Cache:**
* Set-based membership testing
* Updated on new ingredient creation
* No explicit invalidation (append-only)

* Health check endpoints for orchestration

7.2 ENVIRONMENT CONFIGURATION

Configuration managed through:
* `.env` file: Environment variables
* `config.py`: Configuration class with defaults
* Separate DevelopmentConfig and ProductionConfig classes

**Critical Environment Variables:**
* ENVIRONMENT: "dev" or "prod"
* AUTH_SERVER_PORT: Authentication service port
* MAIN_SERVER_PORT: Application service port
* FRONTEND_BASE_URL: Frontend URL for redirects
* SERVER_BASE_URL: API base URL
* JWT_SECRET_KEY: Token signing key
* MONGO_URI: Database connection string
* REDIS_HOST, REDIS_PORT: Cache configuration
* MAIL_SERVER, MAIL_PORT: Email service configuration

7.3 SERVICE HEALTH AND RELIABILITY

**Health Checks:**
* Base endpoints return service status
* Used by orchestration for restart decisions
* Lightweight implementation (10 req/min limit)

**Graceful Degradation:**
* 3-attempt retry logic for AI service
* Fallback error messages for user display
* Redis fallback for rate limiting
* MongoDB connection pooling

7.4 ERROR RECOVERY

**Database Connection Failures:**
* Application startup fails with clear logging
* Container restart attempted by Docker

**Redis Connection Failures:**
* Rate limiting unavailable (all requests limited)
* Cache unavailable (recipes not temporary stored)

**External AI Service Failures:**
* 3 retry attempts with backoff
* Exception raised after retries exhausted
* User receives error message

8.1 DESIGN PATTERNS

**Factory Pattern:**
* Flask app creation through get_app() function
* Configuration object injection
* Blueprint and namespace factory

**Dependency Injection:**
* Configuration passed as parameter
* Extensions initialized per instance
* Loose coupling between components

**Service Locator:**
* Redis and database accessed through global imports
* Extensions initialized at startup
* Available throughout request lifecycle

8.2 CODE ORGANIZATION

**Modular Structure:**
* Routes organized by domain (auth, recipes)
* Models separated from routes
* Shared utilities in helpers module

**Blueprint Organization:**
* Each route group is a blueprint
* Blueprints registered with prefixes
* Namespace organization for RESTX

**Separation of Concerns:**
* Models define data structure
* Routes define API endpoints
* Extensions configure infrastructure
* Errors provide consistent handling
